High Score Leaderboard

Mar 29, 2021
Photo by Element5 Digital on Pexels

Overview

Each team (4 students) was tasked with building a REST API from scratch based on a sample business requirement document (BRD). Teams were given eight weeks with two week sprints.

My Role

In addition to being the SCRUM master, I created automated integration testing workflows and set-up our service architecture in our AWS accounts.

Problem Statement

Currenty each game created by the games team has to implement score tracking functionality individually, leading to manual churn. This also restricts customers from being able to view their game progress across all their games. The REST API must be 1. a consistent mechanism for the games team to track high scores across games and 2. a way for customers to view their high scores across multiple games in one place.

User Stories

As a game developer…

  • I want to submit a score for a given user
    • Acceptance Criteria:
      • The API must accept the userâ€™s account username
      • The API must accept the name of the game the high score was achieved
      • The API must accept the score value of the user
      • The service must store the date the score was submitted
      • The service must store the updated high scores for the user per game
  • I want to track the top scores for a given game
    • Acceptance Criteria:
      • The service must store the name of the game the high score was made
      • The service must store the score value of the user
      • The service must store the username of the player
      • The service must store the date the score was submitted
      • The service must return the scores in descending order
  • I want to allow users to download their high scores across games
    • Acceptance Criteria:
      • The API must return a URL to a JSON file stored in S3
  • I want to allow users to download the top 10 scores for a single game
    • Acceptance Criteria:
      • The API must return a URL to a JSON file stored in S3

Out of Scope

Key features out of scope include:

  • a front end
  • API integration with other Amazon teams
  • username authentication

Architecture

Built with Amazon API Gateway and AWS Lambda, the back-end REST API consisted of 6 endpoints coded with Java:

  • GetUserScores
  • AddUserScore
  • GetGameScores
  • GetUserHighestScores
  • DeleteUserScore
  • GetIdentity

Table Design

Our service used 1 DynamoDB table and 1 global secondary index (GSI).

Scores

  • userName // partition key, string
  • timestamp // sort key, string
  • gameName // string
  • score // number

TopScoresForGameIndex

  • projection: ALL
  • gameName // hash key
  • score // story key

Alternatives

1. AWS Fargate vs AWS Lambda

Fargate does not have Lambda’s cold-start issue and allows longer connection persistence which would have been beneficial if either 1. customers expected to see their high-scores immediately (within 5 seconds of their request) or 2. their scores required 10+ minutes to retrieve/save in DynamoDB. Given the rarity of both situations, we could not justify the added burden of managing Fargateâ€™s dockers.

2. Amazon DynamoDB or Amazon RDS/Aurora

We only needed one table to meet all of our use cases. Therefore, it made sense for us to use DynamoDB (non-relational database) as opposed to Aurora (relational database).
